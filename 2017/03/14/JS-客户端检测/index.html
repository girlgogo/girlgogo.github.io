09 章 客户端检测

9.1 能力检测

说明：又称特性检测。目标不是识别特定的浏览器，而是识别浏览器的能力。
基本模式：

特性：

  • 先检测达成目的最常用的特性
  • 必须检测实际要用到的特性

例子：IE5.0 之前的版本不支持 document.getElementsById()这个 DOM 方法，但可以使用非标准的 document.all 属性实现相同的目的

9.1.2 能力检测，不是浏览器检测

检测某个或某几个特性并不能确定浏览器，不能错误依赖能力检测来检测浏览器。
如果知道自己的应用程序需要使用某些特定的浏览器特性，那么最好是一次性检测所有相关特性。

9.2 怪癖检测

说明：目标是识别浏览器的特殊行为（bug）。建议在脚本一开始就执行此类检测。
例子1: （IE8-） ，如果某个实例属性与标记为 [[DontEnum]]的某个原型属性同名，那么该实例属性将不会出现在 for-in 循环当中。

例子2：（Safari 3 -），会枚举被隐藏的属性。

9.3 用户代理检测

说明：通过检测用户代理字符窜来确定使用的浏览器。（万不得已时才使用）
在每一次 http 请求过程中，用户代理字符窜是作为响应首部发送的，而且该字符窜可以通过 JavaScript 的 navigator.userAgent 属性访问。普遍用于服务器端。

9.3.1 用户代理字符串的历史

HTTP 规范（包括1.0 和1.1版）明确规定，浏览器应该发送简短的用户代理字符串，指明浏览器的名称和版本。
字符串格式为：标识符／产品版本号

呈现引擎：
1、Gecko：Firefox 的呈现引擎，第一个采用的是 Netscape 6 浏览器

Firefox 4 发布后，简化了这个用户代理字符串
1) 删除了“语言”记号
2) 在浏览器使用强加密（默认设置）时，不显示“加密类型”。
3) “平台”记号从 windows 用户代理字符串中删除了，“操作系统或CPU”中始终都包含 “Windows” 字符串。
4) Gecko 版本号固定为 “Gecko／20100101”。

2、WebKit：2003年，Apple 公司发布 Web 浏览器（Safari），呈现引擎是 WebKit。

Safari 3.0 发布后，新增Version记号，这个变化只在 Safari 中有，在 WebKit 中没有。

9.3.2 用户代理字符串检测技术

1、识别呈现引擎
呈现引擎：IE、Gecko、WebKit 、KHTML 和 Opera
呈现引擎检测顺序：
1) Opera：必须得检测 window.Opera 对象（Opera 5 +），该对象用以保存与浏览器有关的标识信息以及与浏览器直接交互。调用 Version() 方法（Opera 7.6+），可以返回一个表示浏览器版本的字符串。
2) WebKit：WebKit 的用户代理字符串中的 “AppleWebKit” 是唯一的。
3) KHTML：由于 Konqueror 3.1 及更早版本不包含 KHTML 的版本，故而就要使用 Konqueror 的版本来代替。
4) Gecko：Gecko 的版本号不会出现在字符串 “Gecko” 的后面，而是会出现在字符串 “rv:”的后面。
5) IE：IE 的版本号位于字符串 “MSIE” 的后面、一个分号的前面。

2、识别浏览器
大多数情况下，识别了浏览器的呈现引擎就足够了。但是，只有呈现引擎还不能说明存在所需的 JavaScript 功能。Safari 和 chrome 都是使用 WebKit 作为呈现引擎，但是它们的 JavaScript 引擎却不一样。clients.webkit 都会返回非 0 值。

3、识别平台
三大主流平台：
1) Windows
2) Mac
3) Unix（包括各种 Linux）

5、识别移动设备

9.3.4 使用方法

用户代理检测是客户端检测的最后一个选择，适用于下列情况：
1) 不能直接准确的使用能力检测或怪癖检测。例如，某些浏览器实现为了将来功能预留的存根函数。在这种情况下，仅测试相应的函数是否存在还得不到足够的信息。
2) 同一款浏览器在不同的平台下具备不同的能力。这时候，可能就有必要确定浏览器位于哪个平台下。
3) 为了跟踪分析等目的需要知道确切的浏览器。
