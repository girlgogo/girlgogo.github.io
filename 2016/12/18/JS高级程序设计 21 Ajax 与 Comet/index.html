Ajax 与 Comet

发表于 2016-12-18 | 分类于 JavaScript高级程序设计（第3版）

21 Ajax 与 Comet

历史

  • JS 需要通过 Java applet 或 Flash 电影等中间层向服务器发送请求
  • 1998采用 人们采用不同手段实现浏览器与服务器的通信(远程脚本)
  • XHR 出现之前，借助一些 hack 手段实现 Ajax ，大多是使用隐藏的框架阔内嵌框架
  • 2005年 Jesse James Garrett 提出 Ajax（Asynchronous JavaScript+XML）

21.1 XMLHttpRequest对象

IE 中三种不同的 XHR 对象

  • MSXML2.XMLHttp
  • MSXML2.XMLHttp.3.0
  • MXSML2.XMLHttp.6.0

创建XHR

  • 根据MSXML库的情况创建：IE7–
  • 原生XHR对象：IE7+、FF、Opera、Chrome和Safari
  • 兼容所有浏览器

21.1.1 XHR的用法

  • xhr.open()方法
    用途：启动一个请求以备发送
    参数（3）：发送请求的类型，”get”或”post”; 请求的URL；表示是否异步发送请求的布尔值
    注意：
  1. URL相对于执行代码的当前页面（也可以是绝对路径）；
  2. 调用该方法不会真的发送请求，只是启动一个请求以备发送；
  3. 只能向同一个域中使用相同端口和协议的URL发送请求
  • xhr.send()方法
    用途：发送请求
    参数：作为请求主体发送的数据；不发送数据必需传入null
  • 响应
    特点：响应数据会自动填充XHR对象的属性
  1. responseText属性：作为响应主体被返回的文本
  2. reponseXML属性：如果响应的内容类型是”text/xml”或”application/xml”，这个属性中将保存响应数据的XML DOM文档；否则为null
  3. status属性：响应的HTTP状态,（200成功；304资源未修改使用缓存）
  4. statusText属性：HTTP状态的说明,跨浏览器时不太可靠
  • 检查状态码

错误报告204状态码

浏览器 XHR类型 错误识别为
IE AciveX 1223
IE 原生XHR 200
Opera 原生XHR 0
  • xhr.readyState属性：请求/响应过程的当前阶段
xhr.readyState属性值 阶段 说明
0 未初始化 尚未调用open()方法
1 启动 已经调用open()方法，尚未调用send()方法
2 发送 已经调用send()方法，尚未接收到响应
3 接收 已经接收到部分响应
4 完成 已经接收到全部响应数据，而且已经可以在客户端使用了
  • readstatechange事件：readState属性值变化时触发
    注意
  1. 在 open() 之前注册事件；
  2. 没有传入 event 事件对象，因为不需要；
  3. 没有使用 this ，因为在有的浏览器中会发生问题，使用实际的 XHR 对象更加可靠

21.1.2 HTTP头部信息

所有浏览器都会发送的头部信息

  • Accept：浏览器能够处理的内容类型
  • Accept-Charset：浏览器能够显示的字符集
  • Accept-Encoding：浏览器能够处理的压缩编码
  • Accept-Language:浏览器当前设置的语言
  • Connection：浏览器与服务器之间连接的类型
  • Cookie：当前页面设置的任何cookie
  • Host：发送请求的页面所在的域
  • Referer：发送请求的页面的URL。
  • User-Agent：浏览器用户代理字符串

设置发送报文头部信息

  • xhr.setRequestHeader()方法
    参数（2）：头部字段的名称、头部字段的值
    注意：必须在 open 和 send 之间设置头部

取得响应报文的头部信息

  • xhr.getResponseHeader()方法
    用途：取得相应的响应头部信息
    参数：头部名称
  • xhr.getAllResponseHeaders()方法
    用途：取得包含所有头部信息的字符串

21.1.3 GET请求

用途：向服务器查询某些信息，必要时可以将查询字符串参数追加到URL的末尾
注意：

  • 位于传入open方法的URL末尾字符串必需经过正确的编码
  • 所有名/值对必需由&分隔

相关方法

  • encodeURLConent()方法
    用途：对每个名/值对进行编码
    参数：名或者值

不安全的示范

自定义向URL的末尾添加查询字符串的函数

21.1.4 POST请求

用途：用于向服务器发送应该被保存的数据
特点：

  • 将数据作为请求报文的主体提交
  • 可以包含非常多的数据，而且格式不限
  • 比GET消耗更多资源

使用XHR来模仿WEB表单提交

  1. 首先设置表单提交时的内容类型，将Content-Type头部信息设置为applicatiom/x-www-form-urlencoded
  2. 其次以适当的格式创建一个字符串，POST数据格式与查询字符串格式相同
  3. 然后再通过XHR发送到服务器

如果不设置Content-Type头部信息，发送给服务器的数据将不会出现在$_POST超级全局变量中

21.2 XMLHttpRequest 2级

兼容性：所有浏览器都实现了规定的部分

21.2.1 FormData

用途：序列化表单或创建与表单格式相同的数据
优点：不必明确地在XHR对象上设置请求头部
兼容性：Firefox4+、 Safari5+、 Chrome、 Android 3+版webkit

  • 用途一：序列化表单
  • 用途二：创建与表单格式相同的数据
  • 常见应用案例：通过XHR传输表单数据

21.2.2 超时设定

  • xhr.timeout属性：表示请求在等待响应多少毫秒之后就终止
    历史：IE8引入 –> 被收入 XMLHttpRequest2 级规范
    特性：如果在设定的时间内浏览器还没有接收到响应，就会触发timeout事件，进而调用ontimeout事件处理程序
    注意：如果超时，xhr.readState仍然变为4，但此时访问xhr.status会出错，为此要设置try-catch
    兼容性：IE8+

21.2.3 overrideMimeType()方法

历史：FireFox引入 –> 被纳入XMLHttpRequest2级规范
用途：重写XHR响应的MIME类型
注意：必需在send()之前调用才能保证重写响应的MIME类型
兼容性：FireFox Safari4+ Opera10.5 Chrome

21.3 进度事件（Progress Events）

历史：W3C的一个工作草案，最早只针对XHR操作->也被其它API借鉴

21.3.1 load事件（Firefox革新）

历史：FireFox尝试简化异步交互模型，用load事件代替readystatechange事件，不用再检查readyState属性
特点：onload事件程序会接收到一个target属性指向XHR对象实例的event对象
兼容性：Firefox Opera Chrome Safari

注意：

  1. 并非所有浏览器都为这个事件实现了适当的事件对象，因此最好自己创建XHR实例
  2. 只要浏览器接收到服务器的响应，不管其状态如何，都会触发load事件。因次必须检查status属性判断数据是否可用

21.3.2 progress事件（Mozilla革新）

特点：onprogress事件处理程序会接收到一个event对象，其target属性是XHR对象，包含3个额外属性
注意：必需在open()之前添加onprogress事件处理程序

  • lengthComputation属性：进度信息是否可用（布尔值）
  • position属性：已经接收到的字节数
  • totalSize属性：根据Content-Length响应头部确定的预期字节数

21.4 跨源资源共享（XHR的语境下）

背景：跨域安全策略
在默认情况下，XHR对象只能访问与包含它的页面位于同一个域中的资源。
解决方法：CORS（跨资源共享）
使用自定义的HTTP头部让浏览器与服务器进行沟通，从而决定请求或响应应该成功，还是应该失败。
演示

  1. 客户端字在发送的post或get请求头重添加Origin头部，值为请求页面的源信息（协议、域名和端口）
  1. 服务器在响应报文中添加Accept-Control-Allow头部，值和客户端请求报文的Origin头部值相同（设置为’*’使得该资源变为公共资源）

21.4.1 IE对CORS的实现

  • XDR (XDomainRequest)：类似XHR对象，但能实现安全可靠的跨域通信。
    兼容性：IE8+
    CORS：部分实现了W3C的CORS规范。
    XDR与XHR区别
  1. cookie不会随请求发送，也不会随响应返回
  2. 只能设置请求头部信息中的Content-Type字段
  3. 不能访问响应头部信息
  4. 只支持get和post信息
  5. 只能创建异步请求
    安全性
    缓解CSRF(Cross-Site Request Forgery，跨站点请求伪造)和XSS（Cross-Site Scripting，跨站点脚本）问题
    过程
  6. 创建一个XDomainRequest的实例
  7. 调用open()方法：和XHR不同，只接受两个参数（请求的类型，URL）
  8. 调用send()方法
  9. 请求返回触发load事件，响应的数据会保存在responseText属性中

注意：只能访问响应的原始文本，无法确定响应的状态码。有效响应触发load事件，失败触发error事件。

  • error事件
  • abort()方法：终止请求
  • ontimeout事件
  • contentType属性：表示发送数据的格式,是通过XDR影响头部信息的唯一方式

21.4.2 其他浏览器对CORS的实现

  • XMLHttpRequest对象对CORS原生支持

兼容性: Firefox3.5 Safari4+ Chrome ios版Safari Android平台webkit

和XDR比较

  • 可以访问status和statusText属性
  • 支持同步请求

和同源请求比较

  • 不能使用setRequestHeader()设置自定义头部
  • 不能发送和接收cookie
  • 调用getAllResquestHeaders()方法总会返回空字符串

跨域方式：使用标准的XHR对象并在open()方法中传入绝对URL

注意
访问本地资源使用相对URL，跨域时使用绝对URL。避免限制访问头部信息或cookie信息等问题。

21.4.3 Preflighted Requests

说明：CROS支持的透明服务器验证机制
方式：支持使用自定义的头部、GET或POST之外的方法，以及不同类型的主体内容，会发送一个Preflight请求
兼容性：Firefox3.5+ Safari4+ Chrome

  • Preflight请求：请求结束后，结果将按照响应中指定的时间缓存起来。第一次发送这种请求是会多一次HTTP请求
  1. 请求报文头部
头部名 值
Origin 与简单的请求相同
Access-Control-Request-Method 请求自身使用的方法
Access-Control-Request-Headers (可选)自定义的头部信息，多个头部以逗号分隔
  1. 响应报文头部
头部类型 值
Access-Control-Allow-Origin 与简单的请求相同
Access-Control-Allow-Methods 允许的方法，多个方法以逗号分隔
Access-Control-Allow-Headers 允许的头部，多个头部以逗号分隔
Access-Control-Max-Age 应该将这个Preflight请求缓存多长时间（以秒表示）

21.4.4 带凭据的请求

背景：跨域请求不提供凭据（cookie、HTTP认证及客户端SSL证明等）

  • 使请求带凭据: 设置XHR的相关属性
    xhr.withCredentials属性：设置为true可以指定某个请求发送凭据

兼容性：Firefox3.5+ Safari4+ Chrome

  • 使服务器接受带凭据的请求：设置相应报文相关头部

注意：如果响应报文中没有该头部

  • 那么浏览器就不会把响应交给JavaScript
  • xhr.responseText值为空字符串
  • xhr.status的值为0
  • onerror事件处理程序将会触发

21.4.5 跨浏览器的CORS

检查是否存在 withCredenttials 属性，再结合检测 XDomainRequest 对象是否存在，就可以兼顾所有浏览器。

共同的属性和方法（所有浏览器都能正常使用）

  • onerror：用于替代onreadystatechange检测错误
  • onload：用于替代onreadystatechange检测成功
  • responseText：用于取得响应内容
  • abort()：用于停止正在进行的请求
  • send()：用于发送请求

21.5 其它跨域技术

21.5.1 图像Ping：与服务器进行简单、单项的跨域通信的一种方式

背景：一个网页可以从任何网页加载图像，也就是说可以通过设置图像的src属性为目标服务器来向其发送get请求
方式：动态创建图像，侦听load和error事件
用途：跟中用户点击页面或动态广告曝光次数
缺点：只能发送get请求，无法访问服务器的响应文本，只能用于浏览器项服务器的单向跨域

21.5.2 JSONP

原理：在请求中指定回调函数，当响应到达时数据将传入回调函数。
方式：创建动态script元素并为其src属性设置跨域URL和回调函数
优点：简单易用、双向通信，能够访问响应数据
缺点：响应中可能夹带恶意代码被客户端加载执行；难以判断请求是否成功。

21.5.3 Comet(服务器推送)

背景：Ajax是一种从页面向服务器请求数据的技术，而Comet是一种服务器向页面推送数据的技术
特点：将信息近乎实时地推送到页面
用途：比如体育比赛的分数和股票股价
实现方式：长轮询和流

  • 长轮询
    背景：短轮询（浏览器定时向服务器发送请求，无论是否更新服务器立即响应）

    特点：浏览器发送请求，服务器保持连接打开，直到有数据发送时发送响应并关闭连接；浏览器随即发送一个新请求。

  • HTTP流
    背景：所有服务器端语言都支持打印到输出缓存然后刷新，这是实现HTTP流的关键所在
    特点：在页面的整个生命周期中只使用一个HTTP连接，服务器端周期性地向浏览器发送数据
    注意：为简化这一技术，为Comet创建了两个新的接口（下一节）
    兼容性：Firefox Safari Opera Chrome

实现方式：侦听readystatechange事件及检测readyState的值是否为3，可以利用XHR对象实现HTTP流

21.5.4 服务器发送事件

SSE(Server-Sent Events，服务器发送事件)
用途：创建到服务器的单向连接
特点：

  • 可以发送任意数量的数据
  • 支持短轮询、长轮询、和HTTP流
  • 服务器响应的MIME类型必需是text/event-stream
  • 能够在断开连接时自动确定何时重新连接

兼容性：Firefox6+ Safari5+ Opera11+ Chrome ios4+版Safari

  1. SSE API

EventSource对象
参数：URL，必须与创建对象的页面同源
实例属性
source.readyState属性

readyState值 表示
0 正连接到服务器
1 打开了连接
2 关闭了连接

相关事件：

事件 触发
open 在建立连接时触发
message 从服务器接收到新事件时触发
error 在无法建立连接时触发

相关事件对象属性：

  • event.data属性：服务器发回的数据
    方法
  • source.close()：强制立即断开连接并且不再重新连接
  1. 事件流
    含义：服务器通过一个持久的HTTP连接不断发送数据给客户端，继而在客户端不断触发服务器事件的过程。
    前缀：服务器端返回的数据带有的”data:”或”id:”前缀
  • data前缀：”data:”,代表普通的数据
    特点：
  1. 只有在包含”data：”前缀的数据行后面有空行时才会触发客户端的message事件
  2. 连续的几个”data：”前缀的数据，其前缀之后的数据将合并为一个，之间以一个换行符分隔
  • id前缀:”id:”，将会给特定的事件指定一个关联的ID
    特点：
  1. 可以位于data:行前面或者后面
  2. 设置了ID后，EventSource对象会跟踪上一次触发的事件
  3. 如果连接断开，会想服务器发送一个包含名为Last-Event-ID的特殊HTTP头部的请求，以便服务器知道下一次该触发那个事件。

用途：在多次连接的事件流中确保浏览器以正确的顺序收到连接的数据段。

21.5.5 Web Sockets

目标：在一个单独的持久连接上提供全双工、双向通信
工作方式：
（1）JS创建Web Socket
（2）一个HTTP请求发送到浏览器发起连接
（3）服务器响应后将HTTP协议升级为Web Socket协议
注意：

  • 标准的HTTP服务器不支持Web Socket协议
  • 不支持DOM2级事件侦听器，必需使用DOM0级语法分别定义每个事件处理程序

模式：将来可能支持其它模式
加密的连接：ws://
未加密的连接：wss://
兼容性：Firefox6+ Safari5+ Chrome ios4+版Safari

1、Web Sockets API
注意：必需给WebSocket构造函数传入绝对URL；同源策略对WebSocket不适用；通信取决于服务器

WebSocket对象
注意：没有readystatechange事件
属性：

  • readyState属性：表示当前连接的状态
属性值 对应数字 含义
WebSocket.OPENING 0 正在建立连接
WebSocket.OPEN 1 已经建立连接
WebSocket.CLOSING 2 正在关闭连接
WebSocket.CLOSE 3 已经关闭连接

方法

  • socket.close()：在任何时候都可以关闭Web Socket连接
    2、发送和接收数据
  • socket.send()：发送数据，如果不是纯文本数据需要进行序列化
  • message事件：当服务器向客户端发来消息时
    event.data: 服务器发来的数据，字符串，复杂数据结构需要反序列化

21.6 安全

CSRF(Cross-Site Request Forgery，跨站点请求伪造)：为被授权的系统通过伪装自己，有权访问某个资源的情况
有效的方式

  • 要求以SSL连接来访问可以通过XHR请求的资源
  • 要求每一次请求都要附带经过相应算法得到的验证码

无效的方式

  • 要求发送POST而不是GET请求-很容易改变
  • 检查来源URL以确保是否可信-来源记录很容易伪造
  • 基于cookie信息进行验证-同样很容易围在

附：表面安全实际危险
